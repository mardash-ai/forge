## Repository Status

Forge has a **running v1 implementation** alongside the specification. The implementation is a Docker-first, API-first platform (TypeScript, run via `tsx`; Fastify API; filesystem-backed Resource/Event store) that can initialize, provision, install, run, build, test, lint, inspect, and explain failures for a Dockerized Next.js web app. The spec in `docs/` remains the source of truth; code must conform to it.

Source lives under `src/` (see "Implementation Map" below). Generated Builder apps and the Resource/Event store are written under `workspace/` (gitignored) — they are not part of the platform source.

### Running / building / testing

Everything runs in Docker. Host dependencies are **Docker + the Compose plugin** (the platform and all app work run in containers — no local Node/npm is assumed even though `tsx` runs the platform) and the **authenticated GitHub CLI (`gh`)** — releases publish images via GitHub Actions, and `gh` is how you watch publish/CI runs, inspect failed jobs (`gh run watch`, `gh run view --log-failed`), and manage PRs.

```bash
make up        # build image + start the platform API (docker compose up -d --build)
make down      # stop the platform
make logs      # tail platform logs
make shell     # shell into the platform container
make test      # run the platform's OWN vitest suite inside the container
./forge <cmd>  # thin CLI wrapper -> runs the CLI inside the container -> calls the API
```

Builder-facing flow (all Dockerized): `./forge init app --name <n>` → `provision` → `install` → `build` / `test` / `lint` / `dev` → `inspect <type>` → `explain --resource <id>` → `plan --goal "…"`.

### Operational notes that bite

- **The API is a long-running process.** `src/` is bind-mounted and the API runs under `tsx watch`, so editing platform source auto-reloads it. If a change doesn't take effect, `make restart` (or `docker compose up -d api`). A generated app's `compose.yaml` is only regenerated by re-running `forge provision`.
- **Docker-out-of-Docker.** The `api` container mounts the host Docker socket and runs `docker compose` for Builder apps. The workspace is bind-mounted at the **identical absolute path** host==container (`${PWD}/workspace`) so nested compose bind-mounts resolve on the host daemon. Do not break this path symmetry.
- **`next build` must run as production.** Never pin `NODE_ENV=development` in a generated app's compose service — Next sets it per command, and forcing development breaks `next build` prerendering.
- **Adding `tests/` or new top-level dirs** to the image requires updating both the `Dockerfile` COPY lines and (for live iteration) the `compose.yaml` bind mounts.

## Implementation Map

Code is placed by *which concept it defines* (per `06_FORGE_REPOSITORY.md`):

- `src/capabilities/<slug>/` — Capability contracts + `execute` (behavior). Registered in `src/capabilities/index.ts`.
- `src/core/` — the boring, stable center: `runtime.ts` (validate → permission → policy → execute → audit), `registry.ts` (routing/discovery), `log-analyzer.ts` (heuristic ExplainFailure).
- `src/plugins/<impl>/` — technology Implementations ONLY (scaffold-nextjs-npm, runtime-docker-compose, package-npm, build-npm, test-npm, lint-eslint). Provider specifics live here and nowhere else.
- `src/resources/types.ts` — Resource shapes (durable state, no behavior). `src/events/catalog.ts` — Event types (facts).
- `src/policies/`, `src/permissions/` — governance + authorization checks the runtime calls before executing.
- `src/storage/store.ts` — filesystem Resource/Event store (JSON docs + `events.jsonl`).
- `src/api/server.ts` — HTTP surface (Capability/Resource/Event/discovery endpoints). `src/cli/index.ts` — thin API client with token-conscious output (`src/cli/render.ts`).

When adding a Capability, follow the recipe in `06_FORGE_REPOSITORY.md`: define the contract, its Resource, its Events, policy/permission needs, one Implementation (as a plugin only at a real technology boundary), then register it. Keep CLI output compact — every returned token costs money.

## Core Mental Model

The single most important thing to internalize: **Forge rigidly separates the Domain Model (the problem language) from Architecture (the solution structure). Never mix them.**

The **seven domain concepts** are the only canonical vocabulary. Use these exact words:

- **Builder** — any actor that creates software (human, agent, service, or system). Not a job title, not a synonym for "engineer."
- **Goal** — the desired outcome; the source of work. Not a command, not a workflow.
- **Capability** — something Forge can *do* (Build, Test, Deploy, Evaluate, Plan, Generate, Search, Analyze, Observe, Review). Capabilities are behavior.
- **Resource** — durable *state* (Workspace, Build, TestRun, Deployment, Evaluation, Artifact, Release, Incident). Resources have identity/lifecycle/history but **own no behavior**.
- **Event** — an immutable *fact* about what happened (BuildCompleted, DeploymentRolledBack). Events are not commands and carry no intent.
- **Policy** — a governance rule that decides what is allowed/required/blocked/escalated.
- **Permission** — an authorization relationship: which Builder may use which Capability or access which Resource.

Canonical relationship: *Builders pursue Goals through Capabilities that create and evolve Resources, emit Events, obey Policies, and respect Permissions.*

**Architecture-only** terms (use only when describing implementation, never as domain concepts): Implementation, Plugin, API, Workflow Composition, Interface, Runtime.

## Load-Bearing Rules

These come from the Laws, Architecture, and API Philosophy. Violating them is a design error, not a style preference:

- **Behavior lives in Capabilities; state lives in Resources.** Never put behavior into a Resource or state into a Capability.
- **Implementations are replaceable and hidden.** A Capability (e.g. `Build`) is a stable contract; the technology behind it (Bazel, Buck2) is an Implementation. API surfaces expose `Build`/`Deploy`/`Evaluate` — never "Run Bazel" or "Apply Kubernetes YAML."
- **Plugins exist only at technology boundaries** (build systems, test frameworks, deploy targets, model providers, SCM, identity, runtimes). Do **not** create a Plugin for business logic or speculative abstraction.
- **The core stays boring and stable.** It owns capability routing, resource lifecycle, event publication, policy evaluation, permission checks, implementation selection, workflow composition, and audit — and knows nothing about specific providers.
- **Humans and agents use the same contracts.** Agents are Builders, not features. Do not build agent-only or human-only access paths to Capabilities and Resources.
- **Long-running work returns a Resource, not a blocked call** (e.g. `POST /capabilities/build` → `202` + a Build Resource the caller observes via state and Events). Mutating APIs should be idempotent.
- **Workflow composition is explicit architecture, not a domain primitive.** Do not hide critical behavior inside Plugins or inside a workflow engine.

## Intended Repository Layout (per `06_FORGE_REPOSITORY.md`)

When code is added, place it by asking *which concept does this define?* The top-level directories are: `capabilities/`, `resources/`, `events/`, `policies/`, `permissions/`, `goals/`, `core/`, `plugins/`, `interfaces/` (`cli/`, `ui/`, `sdk/`), `agents/`, `docs/`, `tools/`.

- Provider-specific code goes under `plugins/<capability>/<technology>/` — never in `resources/` or in a Capability contract.
- `interfaces/` call Capability and Resource APIs and must **not** bypass the core.
- `agents/` holds only first-party agent behavior; platform capabilities never live inside agents.

`06_FORGE_REPOSITORY.md` contains step-by-step recipes for "Adding a Capability," "Adding a Resource," and "Adding an Implementation" — follow them.

## The Governing Question

Every change — feature, Capability, Resource, Policy, Permission, doc edit, PR — answers one question:

> **Does this help Builders build?**

If yes, continue. If no, remove it. When architecture becomes unclear, return to: *Capabilities expose behavior. Resources represent state. Implementations perform work. Plugins package implementations. Events record facts. Policies govern. Permissions authorize.*
